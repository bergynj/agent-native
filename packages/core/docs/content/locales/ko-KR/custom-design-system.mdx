# 커스텀 디자인 시스템

Agent Native는 회사의 React 컴포넌트로 앱이 소유한 화면과 공유 Toolkit 경험을 렌더링할 수 있습니다. 명시적이고 타입이 지정된 어댑터를 한 번 등록하면 동일한 기능 컨트롤러가 액션, 상태, 접근성 요구 사항, 에이전트에 표시되는 동작을 계속 관리합니다.

어댑터 계약에는 Tailwind, shadcn, Radix, `className` 또는 특정 스타일링 런타임이 필요하지 않습니다. 컴포넌트는 테마 프로바이더, CSS-in-JS, CSS Modules, React Aria 또는 일반 CSS를 사용할 수 있습니다. `className`과 `style`은 선택적인 상호 운용성 훅일 뿐 스타일링 요구 사항이 아닙니다.

완전한 어댑터, 테마 프로바이더 연결, 다크 모드 및 문서화된 v1 제한 사항은 실행 가능한 [Material UI Chat 예제](https://github.com/BuilderIO/agent-native/tree/main/examples/chat-mui)와 [Ant Design Chat 예제](https://github.com/BuilderIO/agent-native/tree/main/examples/chat-antd)를 참고하세요.

## 가장 작은 커스터마이징 단계 선택

| 단계                    | 용도                                                                | 공유되는 것                                   |
| ----------------------- | ------------------------------------------------------------------- | --------------------------------------------- |
| **Theme tokens**        | 기본 표면과 아직 어댑터가 적용되지 않은 표면을 빠르게 브랜드에 맞춤 | Toolkit 컴포넌트, 컨트롤러 및 동작            |
| **Semantic components** | 버튼, 필드, 피커, 다이얼로그, 메뉴, 표면 같은 회사 컨트롤           | 기능 컨트롤러와 제품 계약                     |
| **Product slots**       | 연결 카드나 채팅 기록 레일처럼 한 기능에 대한 회사별 레이아웃       | 기능의 컨트롤러와 액션                        |
| **Eject**               | 지원되는 슬롯으로 표현할 수 없는 제품 동작이나 구조                 | Core 런타임 계약; eject된 기능은 앱 소유가 됨 |

토큰으로 시작하고, 시스템에 대응하는 요소가 있을 때 시맨틱 컴포넌트를 추가하세요. 기능 수준의 조합에는 product slot을 사용하고, 여전히 소유권이 필요한 가장 작은 단위만 eject하세요.

## 어댑터 등록

고객 어댑터는 일반 npm 패키지입니다. `app/design-system.ts`에서 하나를 명시적으로 import하세요. Agent Native는 의존성을 스캔하거나 라이브러리를 자동 감지하거나 JSON에서 컴포넌트를 불러오지 않습니다.

빌드 시 테마를 의존성이 적은 모듈에 두면 Vite가 설정을 로드할 때 고객 React 패키지를 평가할 필요가 없습니다.

```ts filename="app/design-system-theme.ts"
import { defineTheme } from "@agent-native/toolkit/design-system";

export const acmeTheme = defineTheme({
  colors: {
    light: {
      background: "#ffffff",
      foreground: "oklch(24% 0.03 255)",
      primary: "rgb(33 82 255)",
      "primary-foreground": "white",
      border: "hsl(218 20% 88%)",
    },
    dark: {
      background: "#11131a",
      foreground: "#f7f8fb",
      primary: "oklch(72% 0.17 259)",
      "primary-foreground": "#07102f",
    },
  },
  radius: "0.625rem",
  typography: { fontFamily: "Acme Sans, sans-serif" },
});
```

```tsx filename="app/design-system.ts"
import { defineDesignSystem } from "@agent-native/toolkit/design-system";
import { acmeComponents } from "@acme/agent-native-design-system";

import { acmeTheme } from "@/design-system-theme";

export const designSystem = defineDesignSystem({
  name: "Acme",
  theme: acmeTheme,
  components: acmeComponents,
});
```

앱의 프로바이더 연결 지점을 통해 정의를 연결하세요.

```tsx filename="app/components/ui/toolkit-provider.tsx"
import { ToolkitProvider } from "@agent-native/toolkit/provider";

import { designSystem } from "@/design-system";

export function AppToolkitProvider({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <ToolkitProvider designSystem={designSystem}>{children}</ToolkitProvider>
  );
}
```

부분 어댑터도 지원됩니다. 생략한 컴포넌트는 Agent Native의 기본 어댑터를 사용합니다. 렌더링 중 커스텀 컴포넌트 하나가 예외를 발생시키면 컴포넌트별 오류 경계가 실패를 기록하고 해당 컨트롤에 기본 컴포넌트를 렌더링합니다. 나머지 커스텀 시스템은 계속 활성 상태로 유지됩니다.

두 개의 완전한 Chat 앱이 실제 디자인 시스템을 대상으로 이 계약을 검증합니다. [Material UI 예제](https://github.com/BuilderIO/agent-native/tree/main/examples/chat-mui)는 Emotion, `ThemeProvider`, MUI 오버레이 스택을 사용하고, [Ant Design 예제](https://github.com/BuilderIO/agent-native/tree/main/examples/chat-antd)는 `ConfigProvider`와 Ant Design 폼 및 오버레이 프리미티브를 사용합니다. 두 예제의 `app/design-system.tsx` 파일이 주요 통합 참고 자료이며, 위의 Acme 픽스처는 의도적으로 작게 유지되어 단위 테스트에 사용됩니다.

이전 `components={{ Button }}` 프로바이더 API도 호환되며 가장 낮은 우선순위의 `ActionButton` 어댑터로 취급됩니다. 둘 다 제공되면 `designSystem.components.ActionButton`이 우선하고 개발 빌드에서 경고를 기록합니다.

## 빌드 시 테마 토큰

`defineTheme`은 hex, `rgb()`, `hsl()`, OKLCH를 포함한 표준 CSS 색상을 받습니다. Core Vite 플러그인은 이를 검증하고 Agent Native의 기존 시맨틱 변수로 정규화합니다. 어두운 색상은 토큰별로 밝은 팔레트로 폴백하며 Agent Native가 어두운 팔레트를 자동으로 만들지는 않습니다.

테마를 Vite 플러그인에 전달하세요.

```tsx filename="vite.config.ts"
import { agentNative } from "@agent-native/core/vite";
import { defineConfig } from "vite";

import { acmeTheme } from "./app/design-system-theme";

export default defineConfig({
  plugins: [...agentNative({ designSystemTheme: acmeTheme })],
});
```

개발 환경과 React Router 프로덕션 빌드 모두에 포함되도록 앱 루트에서 가상 스타일시트를 한 번 import합니다.

```tsx filename="app/root.tsx"
import "virtual:agent-native-theme.css";
```

플러그인은 해당 가상 테마 CSS를 빌드 시 생성합니다. 시맨틱 토큰은 scaffold의 fallback 값을 재정의하지만 fallback 전용 토큰은 제거하지 않습니다. 버전 1에서는 요청별 또는 조직별로 테마를 선택하지 않습니다.

## 17가지 시맨틱 계약

이 계약은 DOM 형태나 시각적 구현이 아니라 의미와 동작을 정의하도록 의도되었습니다. 제품 상태를 이식 가능하게 유지해야 하는 곳에서는 값과 열림 상태를 제어합니다.

### 리프 계약

| 컴포넌트       | 필수 계약                           | 중요한 선택적 시맨틱                                                              |
| -------------- | ----------------------------------- | --------------------------------------------------------------------------------- |
| `ActionButton` | 필요한 경우 `children`과 `onPress`  | `intent`, `emphasis`, `size`, `pending`, 아이콘, disabled state, button type, ref |
| `IconButton`   | 접근 가능한 `label`, `icon`         | `ActionButton`과 일치하는 버튼 시맨틱                                             |
| `TextField`    | 제어되는 문자열 `value`, `onChange` | label, description, error, input type/mode, validation, adornments, events, ref   |
| `TextArea`     | 제어되는 문자열 `value`, `onChange` | label, description, error, rows, validation, events, ref                          |
| `Spinner`      | 진행률 표시기                       | 접근 가능한 label과 시맨틱 크기                                                   |
| `Skeleton`     | 상호작용하지 않는 플레이스홀더      | width, height, line/rectangle/circle shape                                        |
| `Status`       | `children`                          | neutral/info/success/warning/danger 톤, icon, size                                |
| `Surface`      | `children`                          | semantic element, elevation, padding, interactive press behavior                  |
| `Avatar`       | 접근 가능한 `name`                  | image source, fallback, size, presence status, image ref                          |

### 동작 계약

| 컴포넌트   | 필수 계약                                        | 어댑터가 소유하는 동작                                                          |
| ---------- | ------------------------------------------------ | ------------------------------------------------------------------------------- |
| `Tooltip`  | trigger와 content                                | 제어/비제어 open state, delay, placement, portal behavior                       |
| `Menu`     | trigger, items/sections, `onAction`              | 키보드 탐색, 중첩 항목, selection, dismissal, portal behavior                   |
| `Popover`  | trigger와 children                               | 제어/비제어 open state, dismissal, focus, modality, portal behavior             |
| `Dialog`   | 제어되는 `open`, `onOpenChange`, title, children | dialog semantics, dismissal, initial/restore focus, portal 및 stacking behavior |
| `Picker`   | `mode`, options, 제어되는 `value`, `onChange`    | select/combobox semantics, search/open state, validation, loading/empty state   |
| `Checkbox` | 제어되는 `checked`, `onChange`                   | checkbox semantics, indeterminate/required/invalid state                        |
| `Switch`   | 제어되는 `checked`, `onChange`                   | switch semantics와 disabled state                                               |
| `Tabs`     | items, 제어되는 `value`, `onChange`              | tab semantics, orientation, automatic/manual activation                         |

`intent`는 `primary`, `neutral`, `danger` 중 하나이고 `emphasis`는 `solid`, `outline`, `ghost` 중 하나입니다. 크기는 `compact`, `default`, `large` 중 하나입니다. 이 값들은 목적을 설명하며 기본 UI의 클래스나 variant 이름을 설명하지 않습니다.

기본 어댑터는 이 시각적 props에 시맨틱 속성뿐 아니라 관찰 가능한 동작도 제공합니다. button, spinner, status, skeleton, avatar, dialog 크기는 구체적인 variant를 선택하고, skeleton shape와 avatar presence status는 표현 방식을 바꾸며, 메뉴 선택과 `closeOnAction`은 동작에 영향을 줍니다. 커스텀 어댑터는 같은 의미를 자체 시각적 용어로 매핑할 책임이 있습니다.

`Picker`는 select와 combobox 동작만 다룹니다. 날짜 선택은 버전 1 시맨틱 계약에 포함되지 않습니다.

## 오버레이, 포털 및 포커스

동작 컴포넌트는 디자인 시스템의 오버레이와 포커스 구현을 그대로 사용할 수 있습니다. Radix로 감쌀 필요가 없습니다. 제어되는 상태, 접근 가능한 역할, 제공된 `portalContainer`, `initialFocusRef`, `restoreFocusRef` 연결 지점을 포함한 시맨틱 계약을 준수해야 합니다.

Radix가 호스팅하는 Toolkit 표면에서 네이티브 디자인 시스템 다이얼로그가 열릴 때 이것이 중요합니다. 어댑터는 자체 오버레이 스택을 소유하고, 연결 지점을 통해 두 시스템이 포털 계층, z-index 순서, 초기 포커스, 포커스 복원에 합의할 수 있습니다. 중첩 오버레이와 인접 오버레이는 격리된 컴포넌트 탐색기뿐 아니라 실제 앱에서 테스트하세요.

## 헤드리스 기능 컨트롤러와 제품 슬롯

시맨틱 컴포넌트는 공유 어휘입니다. 기능 컨트롤러는 동작의 경계입니다. 컨트롤러가 상태 전이, 액션, 추적, 뷰 모델을 소유하고 기본 뷰와 제품 슬롯이 동일한 컨트롤러를 사용합니다.

```tsx
<BuilderConnectCard
  className="connection-card"
  render={({ viewModel, className }) => (
    <AcmeConnectionPanel
      className={className}
      title={viewModel.title}
      status={viewModel.status}
      pending={viewModel.action?.pending ?? false}
      onConnect={viewModel.action?.onPress}
    />
  )}
/>
```

이는 기능 수준의 headless UI이지 모든 위젯을 위한 저수준 훅을 두 번째로 제공하는 것이 아닙니다. `BuilderConnectCard`와 `ChatHistoryRail`이 첫 번째 검증 슬라이스이고 settings와 sharing은 같은 controller/default-view/custom-view 불변식을 따릅니다.

`ChatHistoryRail`은 v1에서 의도적으로 불투명한 `renderRail` 슬롯 하나를 유지합니다. 이 슬롯은 레일 뷰 전체를 대체하고 공유 컨트롤러와 list props를 받습니다. 별도의 header, footer, row 슬롯은 두 번째 제품이 필요로 할 때까지 미룹니다. 이 방식으로 API의 범위를 제한하면서 기록 상태와 액션은 공유 컨트롤러에 유지합니다.

슬롯으로 필요한 제품 변경을 표현할 수 없다면 가장 작은 공개 단위를 조사하고 eject하세요.

```bash
agent-native eject --list
agent-native eject inspect toolkit/design-system
agent-native eject toolkit/design-system --app my-app --apply
```

Ejection은 소유권을 이전하지만 Core auth, actions, persistence, access checks, chat transport 또는 agent execution을 포크하지 않습니다.

## 적합성 키트 실행

고객 패키지에서 완전한 컴포넌트 맵을 export하고 브라우저와 유사한 테스트 환경에서 제공된 테스트 키트를 실행하세요.

```tsx filename="design-system.conformance.spec.tsx"
import { describe, it } from "vitest";
import { assertDesignSystemConformance } from "@agent-native/toolkit/conformance";
import { DESIGN_SYSTEM_CONTRACT_VERSION } from "@agent-native/toolkit/design-system";
import { acmeComponents } from "@acme/agent-native-design-system";

describe("Acme Agent Native adapter", () => {
  it("conforms", async () => {
    await assertDesignSystemConformance({
      adapterName: "Acme",
      components: acmeComponents,
      contractVersion: DESIGN_SYSTEM_CONTRACT_VERSION,
    });
  });
});
```

이 스위트는 17개 export 전체, 제어되는 값과 열림 상태, 접근 가능한 역할, 키보드 상호작용, 선택, 포털 대상, z-index 상호 운용성, 초기 포커스 및 포커스 복원을 확인합니다. 어댑터 패키지의 CI에서 실행하고 프로바이더별 조합을 위한 작은 앱 수준 오버레이 테스트를 유지하세요.

계약 변경은 시맨틱 버저닝을 따릅니다.

- 새 컴포넌트와 선택적 props는 호환 가능한 minor 변경입니다.
- 새 필수 props, 삭제된 컴포넌트/props 또는 동작 계약 변경에는 새 contract major가 필요합니다.
- 어댑터의 contract major와 Toolkit의 contract major가 다르면 적합성 키트가 실패합니다.

## 어댑터 레시피

### MUI 스타일 테마 프로바이더 시스템

[Chat + Material UI 예제](https://github.com/BuilderIO/agent-native/tree/main/examples/chat-mui)가 주요 구현 참고 자료입니다. 단일 `app/design-system.tsx` 모듈은 시맨틱 어댑터, MUI light/dark `createTheme` 값, `defineTheme` 토큰 소스를 export합니다. 앱의 `AppToolkitProvider`는 공유 Toolkit 표면 위에 MUI `ThemeProvider`를 유지하고 `vite.config.ts`는 같은 `theme` 값을 빌드 시 CSS 생성에 import합니다. 이를 통해 Emotion 스타일, MUI 포털 및 Agent Native 토큰이 하나의 source of truth를 사용합니다.

회사 테마 프로바이더를 `AppToolkitProvider` 위에 두고 패키지 경계에서 시맨틱 props를 어댑트하세요. `intent`, `emphasis`, `size`를 어댑터 안에서 시스템 고유의 용어로 매핑하고 Agent Native 기능 코드에서는 매핑하지 마세요. 오버레이에서는 `portalContainer`를 디자인 시스템의 container 또는 portal prop으로 전달하고 해당 포커스 매니저로 `initialFocusRef`/`restoreFocusRef`를 구현하세요.

```tsx
export const ActionButton: DesignSystemComponents["ActionButton"] = ({
  intent = "neutral",
  emphasis = "solid",
  pending,
  onPress,
  children,
  ...props
}) => (
  <CompanyButton
    {...props}
    color={intent === "danger" ? "error" : intent}
    variant={emphasis === "solid" ? "contained" : emphasis}
    loading={pending}
    onClick={onPress}
  >
    {children}
  </CompanyButton>
);
```

기능의 테마 프로바이더를 각 어댑터 컴포넌트에 복사하지 마세요. 앱 경계에 한 번 설치해 포털과 오버레이가 동일한 컨텍스트를 상속하도록 하세요.

[Chat MUI 어댑터](https://github.com/BuilderIO/agent-native/blob/main/examples/chat-mui/app/design-system.tsx)가 구현 예제입니다. 하나의 `defineTheme` 값을 export하고, 그 값을 `defineDesignSystem`으로 import하며, 동일한 모듈을 `vite.config.ts`에서도 import합니다. 이 단일 source of truth는 런타임 테마 생성 없이 MUI light/dark 프로바이더 토큰과 빌드 시 시맨틱 CSS 토큰을 일치시킵니다.

### React Aria 기반 시스템

React Aria 컴포넌트나 훅으로 시맨틱을 직접 구현하세요. 제어되는 `value`/`open` props를 유지하고 두 번째 상태 저장소를 만들지 말고 변경 콜백을 전달하세요. 동작 계약에는 오버레이 프로바이더와 포커스 범위를 사용하고 `portalContainer`에서 포털 루트를 설정하며 initial/restore ref를 포커스 관리 API에 연결하세요. 적합성 스위트는 최소 동작 검사이며 디자인 시스템은 전체 키보드 및 스크린 리더 테스트 매트릭스를 계속 책임집니다.

```tsx
export const Checkbox: DesignSystemComponents["Checkbox"] = ({
  checked,
  onChange,
  label,
  ...props
}) => (
  <AriaCheckbox {...props} isSelected={checked} onChange={onChange}>
    {label}
  </AriaCheckbox>
);
```

### CSS Modules와 일반 React

어댑터는 일반 React와 CSS Modules로 구성할 수 있습니다. `className`이 컴포넌트의 스타일 계약을 우회한다면 완전히 무시하세요. 이는 선택 사항입니다. DOM 역할, 접근 가능한 이름, 키보드 동작, 제어되는 상태를 명시적으로 유지하세요.

```tsx
import styles from "./action-button.module.css";

export const ActionButton: DesignSystemComponents["ActionButton"] = ({
  intent = "neutral",
  emphasis = "solid",
  pending,
  disabled,
  onPress,
  children,
  elementRef,
}) => (
  <button
    ref={elementRef}
    className={styles[`${intent}-${emphasis}`]}
    disabled={disabled || pending}
    aria-busy={pending || undefined}
    onClick={onPress}
  >
    {children}
  </button>
);
```

## Version 1 경계

버전 1의 시맨틱 브리지는 모든 서드파티 UI 생태계를 대체하지 않습니다. 다음 표면은 가능한 경우 테마 토큰과 기능 슬롯을 제공하지만 내부 위젯은 아직 시맨틱 어댑터 계약이 아닙니다.

- `@assistant-ui/react` 채팅 및 컴포저 내부
- Tiptap 에디터 키트
- `cmdk`, `vaul`, Recharts, `react-day-picker`
- date-picker 컴포넌트 (`Picker` 계약은 select/combobox만 지원)
- Dispatch의 독립적인 프리미티브 포크
- Pinpoint의 Solid.js 및 Shadow DOM UI

이 경계를 명시하면 어댑터 패키지가 정확한 지원 범위를 주장할 수 있습니다. 동일한 의미와 동작이 둘 이상의 기능에 유용할 때만 새 시맨틱 컴포넌트를 제안하고, 한 기능에만 필요한 차이에는 product slot이나 eject를 사용하세요.

## Chat 예제 지원 범위

실행 가능한 [Material UI Chat](https://github.com/BuilderIO/agent-native/tree/main/examples/chat-mui) 및 [Ant Design Chat](https://github.com/BuilderIO/agent-native/tree/main/examples/chat-antd) 예제는 17개 contract components를 모두 등록하고 어댑터를 통해 16/17을 렌더링합니다. `Menu`가 명시적인 미지원 항목으로 남는 이유는 agent-panel header에 v1 data-only contract로는 표현할 수 없는 live compound submenu가 포함되기 때문입니다. assistant-ui composer/message renderer, cmdk internals, Tiptap editor는 v1 non-goals로 남습니다.

일상적인 Chat 공유 버튼은 여전히 기존 popover입니다. 제공되는 `/design-system-proof` 경로는 해당 popover를 디자인 시스템 dialog로 오해하게 만들지 않으면서 Core의 전체 `ShareDialog`에 접근할 수 있게 합니다.
